# Website Decision Memory

## 2026-05-31 - Site Revamp Direction

- The public site should now foreground the current project reality, especially `Manna Minecraft Builder`, instead of presenting the older site as mostly a domain/Manna-View/Landscape shell.
- Added `/projects/manna-minecraft-builder/` as the newest detailed route because the project has enough real working behavior, reference study, visual feedback, and architecture direction to justify public context.
- `Manna Archive` is important enough to appear in the project hub, but it should not get a dedicated route until a few approved posts establish the visual language and posting rhythm.
- `Manna-Landscape` should stay customer-facing. Keep internal Manna system language light there so homeowners see a clear service company, not an internal build diary.
- The site should keep using a generated hero image / real visual asset for first-viewport signal. Do not drift back into only cards, gradients, and text.

- `manna-core.dev` is the umbrella home for the broader manna ecosystem, not only a `Manna-View`
  landing page anymore.
- public profile is not the same as public release; status language should stay explicit.
- `Manna-Landscape` should keep reading like a customer-facing business surface, not like a sibling
  app-marketing page.
- until there is a stronger real photo library, `Manna-Landscape` should avoid thin gallery
  sections and lean on structure, trust cues, and a strong quote flow instead.
- do not invent fake social proof, inflated claims, or generic contractor hype just to make the
  page feel more successful.
- the shared site style should feel aligned with the `manna` design DNA: polished, tactical,
  slightly futuristic, and not generic SaaS.
- detailed project pages should be added selectively where they clarify the network, not because
  every internal build needs its own hype page.
